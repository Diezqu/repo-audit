## D1 · 为什么自建引擎，而不直接用 Claude Code 这类通用 Agent？

**背景**：Claude Code / 通用 Agent 本来就能读本地文件 + 搜网页 + 给购买建议，日常我就是这么用的。那这个项目存在的意义是什么？

**关键区别**：通用 Agent 的控制流是**模型即兴决定的**——这次它可能搜了三个来源，下次可能一个都不搜；核不核实、引不引用，全看模型当时的发挥。自建引擎把控制流**写死在状态机里**：每次必拆解、必并行检索、必三票核实、每条结论必带来源——「有时表现很好」和「每次都结构一致、可核验」是两种东西。控制流固定带来三个通用 Agent 给不了的能力：

1. **可评测**：流程固定才能建评测集、跑回归 CI、给出「引用错误率下降 Z%」这类数字；对聊天记录没法做 ablation。
2. **可优化成本**：分级路由让跑量的活走便宜档，单报告成本可控可归因；通用 Agent 全程旗舰模型。
3. **可交付**：引擎是能开源、能被别人部署的产品；「我的 Claude Code 用法」不是。

**同时也回答了「市面上很多 Agent 应用是不是本质上通用 Agent 也能做」**：功能上多数确实能，垂直 Agent 产品的价值恰恰在工程化那一层——确定性流程、评测、成本、权限、嵌入产品的形态。而这一层正是本项目要展示的能力。

**面试一句话**：Claude Code 能做到一次好的调研，但保证不了每次都是同样结构的调研；我做的是把「有时很好」工程化成「每次都可核验、可评测、可算账」——顺便，通用 Agent 本身就是我想去做的那类系统，用它和造它是两回事。

## D2 · 编排框架：LangGraph，而非 CrewAI / AutoGen / 裸写 asyncio

**备选**：CrewAI（角色扮演式，封装高）、AutoGen（对话驱动）、裸写 asyncio（无依赖）。

**选 LangGraph 的理由**：它把流程建模成**显式状态机**（节点 + 边 + 共享状态），换来三样硬能力：断点续跑（checkpointing 是一等公民）、逐节点可测试（每个节点是纯函数）、并行/合并/失败接管有原生语义（Send 扇出、reducer 合并）。CrewAI 上手快但控制流黑盒，出问题看不进去；裸写 asyncio 等于自己重造这些轮子。2026 年它也是生产采用最广的编排框架（Uber/LinkedIn/NVIDIA 等在用，v1.0 已发布）——选它同时是选生态和岗位 JD 的命中率。

**面试一句话**：我要的三样东西——断点续跑、逐节点测试、显式并行语义——只有显式状态机给得干净，所以 LangGraph；CrewAI 的封装在 demo 阶段是加速器，在排障阶段是黑盒。

## D3 · 证据（Evidence）是结构化对象，不是纯文本

**备选**：每条证据存一段字符串（最简）vs 结构化 `{claim, source_url, excerpt, worker_id}`。

**选结构化的理由**：本项目卖点是「带引用、可核验」，两个下游硬性依赖来源字段——Verifier 要数「这条结论有几个独立来源」来计票，报告要给每条论据挂引用链接。纯文本里抠不出这些，来源是主链路不是装饰，所以第一天就长在数据结构里。`worker_id` 字段则是为排障：哪条证据出问题能追到是哪个 Worker 带回来的。

**面试一句话**：数据结构跟着核心卖点走——可核验是卖点，来源就必须是字段而不是文本里的一句话。

## D4 · 双档模型路由（便宜档 Worker + 旗舰档 Planner/合成器）

**备选**：全用一个模型（简单）vs 按节点分档。

**分档理由**：Worker 是跑量的活（N 个并行、每个只答一个子问题），用便宜档；Planner 的拆解质量决定全局、合成器是最终门面，用旗舰档。成本结构上 Worker 调用次数是 Planner 的 N 倍，把量大的部分降档，单报告成本可以降一个量级（精确数字待 M3 Langfuse 归因后回填）。全部走环境变量配置，换供应商不改代码。

**面试一句话**：把「调用量大」和「质量敏感」两个维度拆开，量大的降档、敏感的保旗舰——成本优化的本质是别让所有 token 都付旗舰价。

## D5 · 代码仓库建在个人知识库之外

**备选**：项目嵌在 Obsidian 库里（管理方便）vs 独立仓库 + MCP 远程读库。

**独立的理由**：仓库要公开上 GitHub，个人库里是身体数据、消费记录、日计划——隐私红线。物理分离让泄露在结构上不可能，而不是靠每次 push 时小心。架构上也更对：引擎通过 MCP 连接器**只读**访问私有数据源，正是真实企业里「应用系统 ↔ 私有数据」的接口化关系。开源边界由此清晰：引擎开源、数据永不出库。

**面试一句话**：引擎是公开产品、库是私有数据，边界必须长在目录结构上而不是长在小心上——我卖的是引擎，不是我的数据。

## D6 · 无密钥「假数据模式」兜底

**备选**：没配 API key 直接报错退出 vs 自动退回假数据占位跑通全流程。

**选兜底的理由**：三个受益方——CI 不需要在 GitHub 上存密钥就能跑端到端回归；任何人 clone 下来零配置先看到骨架跑起来（降低 5 分钟上手门槛）；开发时改图结构不用每次烧真 token。实现上就是配置层给 `try_*` 变体返回 None、节点据此走占位分支，成本极低。

**面试一句话**：让系统在没有外部依赖时仍能跑通骨架，测试的就是纯编排逻辑本身——这也是「逐节点可测试」承诺的兑现。

## D7 · 数据源边界：只用开放网页，有意识放弃小红书/抖音等封闭平台（问题由本人提出）

**背景**：消费测评大量沉淀在小红书/抖音（图文视频、无公开 API、反爬），且充斥软文。要不要抓？

**决定**：不抓。理由：封闭平台采集=另开一个爬虫+多模态项目，高维护、合规灰色，且与本项目要展示的核心能力（编排/核验/评测）正交。数据源声明为搜索引擎可及的开放网页，局限性在 README 与报告中明示——系统在证据不足时输出「存疑/证据不足」而非硬答（首份真实报告已体现此行为）。**软文问题不放弃**：它是 Verifier 的本职——来源独立性计票、跨来源同措辞复读降权、只有优点无缺点的证据降级、结论 <2 独立来源标「存疑」；效果不承诺完美，用 ablation 量化（开/关 Verifier 的引用错误率差）。

**面试一句话**：我明确知道系统看不见什么，并让它在看不见时说「不知道」——数据边界是声明出来的，核验质量是量化出来的。

## D8 · 场景边界与命名：消费/选品决策，不是人生决策（问题由本人提出）

**背景**：「决策调研引擎」字面可以指任何决策。实际场景是什么？名字会不会过度承诺？

**决定**：场景钉死为**重大消费/选品决策**（可调研、证据长在网上、买错有真实代价），不做人生决策（证据不可检索，评测无法建集）。仓库名 `decision-engine` 保留，但所有人读的表述（README 标题、简历条目）一律加限定词「消费决策调研核验引擎」——先给准确的小词，比先给大词再收窄可信。

**面试一句话**：场景收窄不是野心小，是评测可行性的前提——「该不该读博」建不了金标，「该不该买这双鞋」可以。

## D9 · 系统边界：决策支持，不是决策替代（问题由本人提出）

**背景**：衣物类决策存在证据不可及的维度——AI 说可以买，穿上身可能就是不好看。怎么办？

**决定**：明确系统职责=**拦证据可及的硬伤**（版型结构、材质缺陷、渠道与价格风险、与个人护栏冲突），不承诺审美判断。审美的最终核验环节是人的试穿，系统的价值是压缩该环节成本（候选 8 → 2）。个人库 MCP 连接器把可硬化的那部分审美硬化：「好不好看」不可核验，「是否符合已验证的风格护栏（黑色系/锥形/贴而不紧）」可核验。边界在 README 与报告中明示：系统输出证据链与风险提示，决策权在人。

**面试一句话**：我的系统不替用户拍板，它把「必后悔」的选项拦在下单前——Kragg 案例里它给出「不买」靠的全是结构性硬伤，没有一条是替用户判断审美。
