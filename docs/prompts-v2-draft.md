## 0. 接线前必须对齐的 5 个契约事实

写提示词绕不开它下游落到什么数据结构、用什么协议出结构化结果。把这些集中列一屏，接线时逐条对：

1. **状态字段会变。** 现在 `graph.py` 的 `State` 是消费版：`question / subtasks: list[str] / evidence / report`。换域后 `subtasks` 要从「字符串列表」升级成「带目标模块 + 问题清单的对象列表」，`evidence` 要换成「带 file:line 引用的 Claim 列表」。**这个改动属于 T5，不在本文动手**——本文只定义提示词瞄准的目标输出形状，接线时照着落成 Pydantic。

2. **结构化输出走 `function_calling`，不是 `json_schema`。** 这是被真故障打出来的（`BATTLE_LOG.md` 2026-07-03）：DeepSeek 的 OpenAI 兼容层没实现新版 `json_schema` response_format，首次真调用就 400。修法是显式 `with_structured_output(Model, method="function_calling")`。**代价**：function_calling 的 schema 约束力比 json_schema 弱（对嵌套、枚举的支持更松），所以下面的目标 schema 我刻意**保持浅、字段少**，并在提示词正文里**把枚举取值再申明一遍**，用提示词补 schema 松掉的那一截。（任务里提到的「D16」在 `DECISIONS.md` 里其实不存在，这条约束的原始出处是 `BATTLE_LOG.md` + `graph.py:77-80` 注释。）

3. **节点↔模型档映射不变。** Planner = 旗舰档（`try_flagship_tier()`），Worker = 便宜档（`try_cheap_tier()`），合成器 = 旗舰档；Verifier 按 README 走中档。任一档拿不到 key 时节点退回**假数据模式**，骨架照跑、CI 不用密钥——提示词不用管这个分支，但要知道它存在（提示词只在真 tier 路径生效）。

4. **Worker 的工具四件套（T3，尚未实现）。** 本文的 Worker 提示词**假定**下列只读工具存在并按此签名调用。T3 实现时须与此对齐，否则接线时改名。四件套里 Planner 用前两个拼「地图」，Worker 用后三个下钻：

   | 工具 | 签名（本文钉定） | 谁用 | 干什么 |
   |---|---|---|---|
   | `repo_stats` | `repo_stats() -> {语言分布, 文件数, 总行数, 顶层目录}` | Planner | 给仓库量个体型 |
   | `repo_tree` | `repo_tree(path=".", depth=2) -> 目录树` | Planner（depth≤2 拼地图）/ Worker（下钻某子树） | 看结构 |
   | `read_file` | `read_file(path, start=None, end=None) -> 带行号文本` | Worker | 精读、拿行号 |
   | `grep_repo` | `grep_repo(pattern, path_glob=None) -> [file:line + 命中行]` | Worker | 定位符号 |

   > 命名对账：README 里写的简称 `tree/grep` = 本文的 `repo_tree/grep_repo`；`read_file/repo_stats` 一致。T3 的**路径白名单 + 输出截断**是工具层自带的安全措施，不是提示词的活——但提示词要写成**能扛住截断**（见 Worker 铁律 6）。

5. **Verifier 还没建（本周晚些做），但 Worker 的输出就是 Verifier 的输入。** 所以 Claim 的 `file / line_start / line_end` 不只是给人看的引用，它们是 **Verifier 回读源码的坐标**。本文把 Claim schema 当成「Worker↔Verifier 契约」来设计，哪怕 Verifier 这周还没落地——这样它一落地就能直接吃 Worker 的产物，不用返工。

## 1. Planner 提示词

### 1.1 输入契约（模板占位符）

Planner 只吃仓库的「地图」，三样东西拼进提示词：

- `{repo_stats}` — 来自 `repo_stats()`：语言分布、文件数、总行数、顶层目录列表。
- `{tree}` — 来自 `repo_tree(".", depth=2)`：**只到第二层**的目录树。
- `{readme_head}` — README 的开头（约前 60 行 / 2000 字）。**可能为空**（有些仓库没 README）——提示词要能在缺 README 时只靠 stats + tree 照拆。

Planner **拿不到任何文件正文**。这是有意的（见【设计理由】）。

### 1.2 目标输出契约（T5 落成 Pydantic；走 function_calling）

```python
class SubTask(BaseModel):
    target_module: str        # 该子任务聚焦的目录/模块/文件（从目录树里挑）
    questions: list[str]      # 1~3 个"读这块代码就能回答"的具体问题
    # v1 先不加 question_type 字段——理由见【设计理由】末段

class Plan(BaseModel):
    subtasks: list[SubTask] = Field(
        description="3~8 个可并行、独立调研的子任务；仓库过小可酌减，不许注水凑数"
    )
```

> 对照消费版 `_Plan(subtasks: list[str])`：从「一串字符串」升成「带目标模块的对象」，因为 Worker 现在要**知道去哪块代码查**，而不是拿着一句话满仓库乱翻。`worker_id` 这类追踪字段由运行时（Send 载荷）注入，**不进模型输出 schema**——沿用现在 `_SourcedFindings`（模型出）与 `Evidence`（运行时装配加 `worker_id`）分离的做法。

### 1.3 提示词全文

```text
你是代码库调研的规划者。你面前只有一个陌生仓库的「地图」——统计信息、浅层目录树、README 开头——你的唯一职责是把「读懂这个仓库」拆成 3~8 个可以并行、独立调研的子任务，交给下游的 Worker 逐个去代码里取证。

严格约束：
- 你只做拆解，不做回答。你看不到文件正文，任何结论都不该由你给出——读代码下结论是 Worker 的活。
- 每个子任务必须自包含：Worker 看不到这个原始全局目标，只拿到你写的这一条，也要能独立照着去查。
- 子任务之间尽量不重叠：它们会被并行分派，重叠等于花两份钱查同一件事。

拆分原则（两条一起用）：
1. 按模块边界拆——优先沿目录树里看得见的模块 / 子包 / 顶层目录切，一个子任务聚焦一块（例：checkpoint 持久化层、graph 编排层、CLI 入口层各一条）。
2. 按问题类型拆——覆盖下面四类，别全堆在"这是什么"上：
   - 功能定位：某模块/文件是干什么的、对外暴露什么接口
   - 机制原理：某能力具体怎么实现（关键数据结构、控制流、核心算法）
   - 边界条件：错误处理、并发、持久化、失败恢复这些容易踩的边界
   - 设计取舍：为什么这么设计、有没有可插拔/开关/降级的点

每个子任务给出两样：
- target_module：该子任务落在哪个目录/模块/文件（从上面的目录树里挑；拿不准就填最相关的顶层目录，别填一个树里没有的路径）
- questions：1~3 个"读这块代码就能回答"的具体问题。要能被 file:行号 回答，别问"介绍一下 X"这种泛问。

数量：目标 3~8 个。仓库很小就少拆几个（宁少毋凑）；很大也别超 8——把同一模块的问题合并成一条。

──────── 仓库地图 ────────
[统计]
{repo_stats}

[目录树 · 深度≤2]
{tree}

[README 开头]（可能为空）
{readme_head}
```

### 1.4 【设计理由】

**为什么先看地图、只看地图，再让 Worker 下钻？**
把「读懂一个大仓库」想成「摸清一栋陌生大楼」。你不会一进门就逐个房间搜——先看楼层导览图（哪层是什么部门），再派人去具体房间。Planner 就是看导览图的人：`repo_stats` 是大楼多大、`repo_tree` 深度 2 是楼层+部门牌、README 开头是物业写的一句话简介。**为什么不给它文件正文？** 三个原因，面试直接说：
- **职责分离**：Planner 的活是「拆」，不是「答」。给它正文它就想自己下结论，越权了；下结论必须由**读了确切代码、能给出 file:line 的 Worker** 来做——这是整个「可核验」卖点的地基。
- **上下文经济**：一个大仓库的正文可能几十万行，全塞进 Planner 会**上下文爆炸**。地图的大小只跟目录数量走（`O(目录数)`），跟代码行数无关——所以再大的仓库，地图都塞得下。这正是项目方案里写的卖点：「大仓库上下文爆炸 → Planner 只看地图 + Worker 分片（是卖点不是缺陷）」。
- **深度为什么卡 2**：Planner 只需要「看见模块边界」就能拆，看不看得见叶子文件不影响拆。更深的结构交给 Worker 用 `repo_tree` 现场下钻。卡 2 是**时间盒 + 上下文**的折中：够拆解、又不让树本身撑爆提示词。（超大仓库的兜底想法：让 Planner 对最大的两三个目录再 +1 层——但那是 v1.1，v1 先固定 depth≤2，不开这个口子。）

**为什么拆分原则是「模块边界 × 问题类型」这两条？**
- 模块边界那条，是为了**让并行真的省钱**。Worker 是便宜档、N 个并行跑，按目录切能让每个 Worker 的 grep/read 集中在一小片、互不重叠；要是按别的维度切，两个 Worker 可能翻同一批文件，白花一份便宜档的钱。
- 问题类型那条（功能/机制/边界/取舍），是为了**逼出深度、且和评测对齐**。只按模块切，模型容易每条都问成「这是干嘛的」（全是功能定位），报告就浮在表面。钉死四类，等于强制它去问机制和边界。而且这四类**正好是评测集 30 题的配比维度**（功能定位 10 / 机制原理 10 / 边界条件 5 / 设计取舍 5）——Planner 按这套拆，产出天然贴着评测要考的东西。

**为什么输出要结构化成 `target_module + questions`，而不是像 v1 那样一串字符串？**
因为下游 Worker 现在要**带着地址去查**。v1 消费版 Worker 是「拿一句话去搜索引擎搜」，一串字符串够了；v2 Worker 是「去仓库某块代码取证」，得先知道**大概去哪块**（target_module），才好把 8 次工具预算花在刀刃上。`questions` 单独拆成清单（而不是揉进一句话），是为了让 Worker 能逐条对着答、逐条判「这条查实了没」。

**为什么数量是 3~8、还允许少于 3？**
下限 3 是逼它别偷懒（一个仓库总不止一块值得看）；上限 8 是**成本闸**——子任务数 = 并行 Worker 数 = 便宜档调用的倍数，放开了会线性烧钱。允许小仓库少于 3，是因为「宁少毋凑」：硬凑到 3 条会逼出注水子任务，Worker 拿到注水任务只会产出注水 Claim，污染下游。**质量下限靠「不许注水」这句话守，不靠数字兜底。**

**为什么先不加 `question_type` 字段？**
本可以让 Planner 给每条子任务打上「功能/机制/边界/取舍」的标签，看着更整齐、还能和评测配比自动对账。先不加，两个理由：① function_calling 对枚举字段的支持比 json_schema 松，多一个枚举字段多一分模型填错/填空的风险，schema 越浅越稳；② 这周是七天冲刺，标签目前没有消费方（Verifier 不看它、报告不按它分组）——**没有消费方的字段就是负债**。等评测阶段真需要按类型对账了再加，属于 v1.1。

## 2. Worker 提示词

### 2.1 输入契约 + 绑定的工具

- 输入：一条 `SubTask`，拆成两个占位符喂进去——`{target_module}` 和 `{questions}`（逐行列出）。Worker **看不到全局目标**，只看这一条（对应 Planner「子任务自包含」的约束）。
- 运行时另注入 `worker_id`（来自 Send 载荷 `WorkerInput`），**不进提示词、不进模型输出**，只在装配 Claim 时挂上去做追踪。
- 绑定的只读工具：`repo_tree` / `read_file` / `grep_repo`（签名见 §0.4）。工具循环怎么跑是 T5 的接线细节，但**推荐用一个终结工具 `submit_claims` 收尾**（理由见【设计理由】末段）。

### 2.2 目标输出契约（T5 落成 Pydantic；走 function_calling）

```python
class Citation(BaseModel):
    file: str          # 仓库相对路径，逐字取自工具输出，禁止臆造
    line_start: int    # 取自 read_file 的行号——没读过就没有行号可填
    line_end: int
    snippet: str       # 从带行号输出里逐字摘录的关键几行（不转述、不改写）

class ClaimDraft(BaseModel):
    statement: str                            # 对某个子问题的一句话具体结论
    status: str                               # 只能是 "supported" 或 "insufficient"
    citations: list[Citation] = []            # supported 时 ≥1 条；insufficient 时留空

class WorkerOutput(BaseModel):
    claims: list[ClaimDraft] = Field(
        description="每个子问题至少对应一条结论；查不实就出 insufficient，不许省略"
    )
```

> 说明：`status` 故意用 `str` 而不是 `Literal[...]`——function_calling 下枚举约束不稳，取值改由提示词正文兜死（belt-and-suspenders）。运行时把 `ClaimDraft` 装配成带 `worker_id` + 子任务引用的 `Claim`（对照消费版 `Evidence` 加 `worker_id` 的做法）。**Worker 的 status 只有两值**（supported / insufficient）；README 里 Verifier 的 supported/refuted/insufficient 是**三值**——差的那个 `refuted` 只有独立核验才判得出，Worker 判不了自己「说错了」，见【拷问清单】#3。

### 2.3 提示词全文

```text
你是代码库调研员。你领到一个子任务，手上有三个只读工具可以探查这个仓库：
- repo_tree(path, depth)：看某个子目录的结构，找该读哪些文件
- grep_repo(pattern, path_glob)：全库正则搜索，返回 file:行号 + 命中行——用它定位符号/关键字
- read_file(path, start, end)：读带行号的文件内容——用它精读命中处的上下文、拿到确切行号

工作方式：先定位，后精读。典型路径 = 先 grep_repo 找到关键符号在哪个 file:line，再 read_file 把那几行前后读全，必要时顺着引用再 grep 一次。

铁律（这几条直接决定整个引擎可不可信，逐条守）：
1. 每条结论都要能落到具体代码位置。凡是 status=supported 的结论，必须带至少一条引用 {file, line_start, line_end, snippet}。
2. snippet 必须从 read_file 的带行号输出里逐字摘录关键几行——原样复制，不转述、不改写、不补全。
3. file 和行号只能来自工具的真实返回。你没用 read_file 读过的地方，就没有行号可填——绝不许凭印象编造 file:line。
4. 查不实就诚实降级：某个问题翻了代码仍拿不到足够证据，就为它单独出一条 status=insufficient 的结论（statement 写清"证据不足：<这个问题>"，citations 留空）。绝不许用你脑子里的先验知识硬答，更不许编一条引用来凑格式。
5. 工具调用上限 8 次，用完即交卷。把预算花在最可能有答案的地方；先定位后精读；到第 6~7 次仍没查实的问题，直接标 insufficient，别为多查一次而放弃收尾。
6. 工具输出可能被截断（大文件、海量命中）。看到截断标记就缩小范围重查（更精确的 pattern、更窄的行区间），别假设你已经看到了全文。
7. 只答这个子任务。与它无关的发现，不管多有意思，都不要塞进来。
8. 你是只读的。绝不描述"我修改了/建议改成"，你的活是取证，不是改代码。

子任务目标模块：{target_module}

要回答的问题：
{questions}

调查完成后，输出结论列表 claims：每条 = statement + status(supported|insufficient) + citations[]。上面每一个问题都要有对应结论，哪怕结论是 insufficient。
```

### 2.4 【设计理由】

**为什么强制每条结论带引用、没引用就得自标 insufficient？**
这是整个项目的**命根子**，一句话就能讲：市面上的代码问答 bot 会很流利地告诉你「这个仓库支持跨进程恢复」，但它**不告诉你这话是从哪几行代码看出来的**——你没法查，它也可能是顺着语感编的。我这套引擎的差异点就是「每句话都能翻回源码对账」。所以：
- **supported 必须带 `{file, line_start, line_end, snippet}`**——这四个字段是给下游 Verifier 的**回读坐标**：Verifier 会真的打开这个 file、翻到这几行、比对 snippet 对不对得上。没有坐标，核验无从谈起。
- **没证据就必须标 insufficient，而不是硬答**——这是把「不知道」变成一等公民。一个诚实说「这块代码没找到答案」的 Worker，比一个用训练记忆流畅编答案的 Worker 有用得多，因为**编出来的答案会污染报告、还骗过评测**。这条直接呼应项目的核心卖点：证据不足时输出「存疑」而不是硬答。

**为什么它防得住模型编 file:line？（面试必问）**
不指望单靠一句「别编」。是**三层防线**叠起来的：
1. **提示层**（铁律 2、3）：snippet 要逐字摘自工具输出、行号只能来自 read_file 真实返回、没读过就没行号可填。把编造的路堵在措辞上。
2. **结构层**：line_start/line_end 是整数字段，值得从工具输出里拿——Worker 要想填一个具体行号，它**得真的调过 read_file** 才拿得到那个数，凭空想一个数字对上真实代码的概率极低。
3. **核验层**：就算前两层漏了，Verifier 回读那个 file:line 做字符串比对，**编的引用对不上，直接判 refuted**，而且这个「编造率」会被 Verifier 开/关的 ablation **量化成简历上的硬数字**。
一句话版：提示词约束 + 独立核验 + 数字兜底，不赌单点。（这也延续了 `BATTLE_LOG` 那条教训——别吃默认、把协议钉死，这里钉死的是「证据协议」。）

**为什么调用上限是 8？太少会不会把 Worker 腰斩、答不全？**
先说 8 怎么来的：一条子任务的典型取证路径是——grep 定位（1~2 次）→ read_file 精读命中的几个文件（2~3 次）→ 顺着一个引用再 grep 一次、再 read 一次（2~3 次），加起来正好落在 8 以内。8 给了「定位—精读—追一层引用」的完整回合，又不至于让模型有预算在整个仓库里漫游。
再说「腰斩」这个担心，答案恰恰是**腰斩不等于胡编**：铁律 5 规定「到点还没查实就标 insufficient」——所以到上限的后果是**诚实降级**，不是硬凑一个没根据的答案。这正是核验体系想要的行为。至于 8 vs 6 vs 12，是个**可调参数**，评测阶段可以回调；真正的护栏不是「8」这个数字，而是**「到点即降级」这条规则**——数字变了规则不变，系统的诚实性不受影响。
上限 8 还有个工程意义：它是**单人项目的可靠性纪律**之一（配合路径白名单、工具输出截断、超时降级）。没有上限，一个跑飞的 Worker 能把便宜档的账单和延迟拖爆——单人维护恰恰需要这种硬闸。

**为什么先定位后精读，而不是让它自由发挥？**
grep 是便宜的（一次搜全库），read_file 是「贵」的（读进来占上下文）。先 grep 把范围缩到几个 file:line，再只 read 那几段，等于**用便宜操作省贵操作**——同样 8 次预算能覆盖更多、看得更准。反过来先大段 read 再找，既费预算又容易把上下文塞满。这跟「Planner 先看地图」是同一个思路的下沉：**永远先用低成本手段缩小范围**。

**为什么输出格式这么约束（两值 status、snippet 逐字、走 function_calling）？**
- **status 只给两值**：Worker 能判的只有「我有引用支撑」或「我没查到」。「说错了」（refuted）得靠一个**没写这条结论、不偏袒**的第三方回读才判得出——那是 Verifier 的活。让 Worker 自己判 refuted 等于让它承认自己错，模型会偏袒自己，判不准。两层状态语义分清，是「Verifier 为什么要独立成层」在数据上的落点。
- **snippet 逐字**：只有逐字复制，Verifier 和人才能拿它跟源码**字符串比对**；一转述就比不了，核验就退化成「看着差不多」。这跟消费版 `Evidence.excerpt`「防转述失真」是同一个考虑。
- **走 function_calling + schema 保持浅**：被 `BATTLE_LOG` 那次 400 教的——DeepSeek 不认 json_schema，只能走 function_calling，而 function_calling 的 schema 约束更松，所以我把 Claim 结构压浅、字段少、枚举取值再在提示词正文里申明一遍。用**提示词的话**补上 **schema 松掉的约束**，换来的是跨供应商（DeepSeek/Qwen 都能跑）的可移植性。

**接线提示：为什么建议用终结工具 `submit_claims` 收尾？**
一个技术坑，值得先讲清：模型「循环调工具查资料」和「最后吐结构化 Claim」**在 function_calling 里是同一条通道**——你没法在一次调用里既 `bind_tools(查资料的三件套)` 又 `with_structured_output(Claim)`。最稳的解法是把「交卷」也做成一个工具 `submit_claims(claims=[...])`，和三个查资料工具一起绑上去：模型查够了就调 `submit_claims` 结束循环。这样**整个 Worker 从头到尾都是统一的工具调用协议**，正好落在 DeepSeek 唯一稳的那条路上。这是 T5 接线时的选择，写在这里免得接线的人再踩一遍 `bind_tools` 和 `structured_output` 打架的坑。

## 3. 【拷问清单】面试官视角 5 问 + 参考答题要点

> 用法：这 5 问是模拟面试官对着上面这套提示词设计发起的追问。答题要点是**弹药**不是背稿，讲的时候用自己的话、能扯到真实取舍/数字最好。

**问 1：Planner 只喂地图不喂文件内容——可大仓库的目录树本身也可能很大，depth≤2 真的够拆？不会漏掉深处的重要模块？**
- 职责分离：Planner 的活是「拆」不是「答」，喂正文会诱它越权下结论，而下结论必须由能给 file:line 的 Worker 来做——这是「可核验」的地基。
- 上下文经济：地图大小跟目录数走（`O(目录数)`），跟代码行数无关，所以再大的仓库地图都塞得下；喂正文才会上下文爆炸。这是**卖点不是缺陷**（方案原文如此）。
- depth≤2 够，是因为 Planner 只需「看见模块边界」就能拆，深处交给 Worker 的 `repo_tree` 现场下钻；卡 2 是时间盒折中。超大仓库的兜底（对最大的几个目录再 +1 层）想清楚了，但划到 v1.1，v1 不开这口子守时间盒。
- 反问式收尾：漏不漏得看**评测**——30 题里就有「机制原理/边界条件」类，若 Planner 拆解漏了深处模块，这些题会掉分，是能被数字抓住的，不是靠嘴保证。

**问 2：工具调用上限 8，凭什么是 8？是不是拍脑袋？会不会把没查完的 Worker 一刀切了？**
- 8 是按典型路径估的：grep 定位(1~2) + read 精读(2~3) + 追一层引用(2~3) ≈ 8，够一个完整「定位—精读—追引用」回合，又不给模型在全仓库漫游的预算。
- 「切了」不等于「胡编」：到点未查实的问题按规则标 insufficient（诚实降级），这正是核验体系要的行为，不是硬凑答案。
- 8 是可调参数，评测阶段可回调；**真正的护栏是「到点即降级」这条规则**，不是数字本身——数字变、规则不变，诚实性不受影响。
- 工程意义：它是单人项目的可靠性硬闸之一（配合路径白名单、输出截断、超时降级），没有它一个跑飞的 Worker 能拖爆便宜档账单和延迟。这条正好是项目「单人怎么保可靠」的标准答案之一。

**问 3：Worker 已经自标 supported/insufficient 了，为什么还要一个独立 Verifier？不是重复劳动吗？**
- 写结论的模型自查会偏袒——让它给自己的话打分，天然倾向说「我对」。独立层才客观。这是 Verifier 独立成层的根本理由。
- 两者语义不同：Worker 的 supported 只是「我自认有引用」；Verifier 是「回到 file:line 独立复核这几行代码**真的**支持这句话吗」——一个是自述，一个是复核。
- Worker 自标是**廉价的上游过滤**：把「根本没证据」的直接挡在 insufficient，Verifier 只需裁「有引用」的那批，省 Verifier 的调用。
- 状态值也对得上这套分工：Worker∈{supported, insufficient} 两值（它判不了自己「说错了」），Verifier∈{supported, refuted, insufficient} 三值，多出来的 refuted 只有独立核验判得出。
- 最后一锤：Verifier **可开关**才有 ablation——同套题开/关各跑一遍得出「引用错误率差」，这是简历上的核心硬数字，独立成层是这个数字的前提。

**问 4：强制带引用，会不会反而逼着模型编 file:line 来凑格式？你怎么防住？**
- 不赌单点，三层防线：① 提示层——snippet 逐字摘自工具输出、行号只能来自 read_file、没读过就没号可填；② 结构层——行号是整数，得真调过 read_file 才拿得到，凭空想一个数对上真实代码概率极低；③ 核验层——Verifier 回读比对，编的对不上直接 refuted。
- 兜底是数字：这个「编造率」被 Verifier 开/关的 ablation 量化，不是靠承诺「不会编」。
- 思想钢印：呼应 `BATTLE_LOG` 那条教训——别吃默认、把协议钉死；这里钉死的是「证据协议」。防幻觉不是靠一句祈使句，是靠**约束 + 核验 + 度量**三件套。

**问 5：结构化输出为什么走 function_calling 而不用约束更强的 json_schema？是不是牺牲了 schema 的约束力？**
- 这是被真故障打出来的，不是选型偏好：DeepSeek 兼容层没实现 json_schema response_format，首次真调用直接 400（`BATTLE_LOG` 2026-07-03）。教训原话——「"OpenAI 兼容"是个程度词，不是布尔值」。
- 确实有代价：function_calling 对嵌套、枚举的约束比 json_schema 松。我的对冲是**把 schema 压浅**（Claim 只三四个字段、status 用 str 不用 Literal）+ **在提示词正文里把取值再申明一遍**，用提示词补上 schema 松掉的那截。
- 取舍口径：跨供应商可移植性（DeepSeek/Qwen 都能跑）> 单一供应商上的强 schema。对一个要在不同便宜档模型间切换的引擎，可移植性更值钱。
- 加分项：这也顺带解释了为什么 Worker 建议用终结工具 `submit_claims` 收尾——既然全程只能靠 function_calling，那把「交卷」也做成工具，整个循环就是一条统一协议，避开 `bind_tools` 与 `structured_output` 打架。
